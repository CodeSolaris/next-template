# 🍰 Feature-Sliced Design in Next.js (2025-2026)

This project strictly follows the [Feature-Sliced Design (FSD)](https://feature-sliced.design/) architectural methodology, adapted for the modern Next.js App Router paradigm.

## 🧱 Layers (Слои) — Сверху вниз
Иерархия слоев строго регламентирована. Компонент может импортировать только из слоев, находящихся ниже него.

- **App:** Глобальная инициализация (провайдеры, стили, стор, шрифты). Содержит конфигурацию всего приложения. Без слайсов.
- **Pages:** Сборка полноценных страниц из виджетов, фич и сущностей. Отражает структуру роутинга.
- **Widgets:** Самостоятельные крупные композиционные блоки (например, `Header`, `ProductGrid`, `Sidebar`).
- **Features:** Интерактивные действия пользователя, приносящие бизнес-ценность (глаголы: `AuthByEmail`, `AddToCart`, `SendFeedback`).
- **Entities:** Структурные бизнес-сущности (существительные: `User`, `Product`, `Order`). Содержат UI отображение сущности и базовые хуки.
- **Shared:** Переиспользуемая инфраструктура (UI-kit, утилиты, API клиенты, конфиги). Без бизнес-логики и без слайсов.

## 🍰 Slices & Segments (Слайсы и Сегменты)
- **Slice (Слайс):** Группировка кода по бизнес-смыслу внутри слоя (например, `entities/user` или `features/auth`).
- **Segment (Сегмент):** Техническое деление логики внутри слайса. Основные сегменты:
  - `ui/`: UI-компоненты и стили (React components).
  - `model/`: Бизнес-логика, типы, Server Actions, стейт-менеджмент.
  - `api/`: Запросы к бэкенду (взаимодействие с внешними API).
  - `lib/`: Хелперы и локальные хуки.

## 🚀 Next.js Best Practices
- **Root `app/` vs `src/`:** В корневой директории `app/` находится только "тонкий" роутинг. Вся реальная логика и UI лежат в слоях внутри `src/`.
- **Proxy Pattern:** Маршруты в `app/` просто реэкспортируют страницы из слоя `pages`:
  ```tsx
  // app/home/page.tsx
  export { HomePage as default } from '@/pages/home'
  ```
- **Server Actions:** Рекомендуется хранить в `features/[slice]/model/actions.ts` или аналогичном месте в `entities`. Не используйте их для простого чтения данных (для этого есть обычные `await fetch` или ORM запросы в Server Components).

## 🛡️ Import Rules (Главные запреты)
- 🚫 **Запрещен импорт сквозь слои вверх:** Слой `entities` никогда не может импортировать из `features` или `widgets`.
- 🚫 **Запрещен кросс-импорт слайсов:** Например, слайс `entities/user` не может импортировать напрямую из слайса `entities/product`. Для связи используйте `features` или `widgets`.
- 🔒 **Инкапсуляция:** Внешнее взаимодействие со слайсом происходит ТОЛЬКО через его Public API (файл `index.ts` в корне слайса). Нельзя импортировать внутренние файлы напрямую (например, `entities/user/ui/user-card` — ошибка, правильно: `entities/user`).

## 🛠 Tools
В проекте настроены инструменты для контроля архитектуры:
- **Steiger:** Официальный линтер структуры файлов и директорий для FSD. Проверяет наличие папок и Public API.
- **ESLint (`eslint-plugin-boundaries`):** Контролирует правила импорта между архитектурными слоями и слайсами.
- **fsd-cli:** Используйте для быстрой генерации готовых слайсов со всеми нужными сегментами.

## 📝 Naming Conventions
- **Папки и Файлы:** `kebab-case` (безопасный формат для кросс-платформенной работы Windows/Linux). Пример: `login-form.tsx`.
- **React Компоненты и Типы:** `PascalCase`. Пример: `UserProfile`, `AuthResult`.
- **Переменные и Хуки:** `camelCase`. Пример: `userData`, `useAuth()`.
- **Shared UI:** Компоненты базового UI-кита принято группировать или префиксировать (зависит от настроек).
